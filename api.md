## 2. Talk to Data — `POST /api/talk-to-data`

Main endpoint. Accepts a natural language question together with a JSON-LD dataset and an ontology URL. Returns a differentially private, natural language answer along with session metadata, remaining privacy budget, and the noisy query results.

### Request Body

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `question` | `string` | **Yes** | Natural language question to ask about the data. Required even if `adjusted_query` is provided for context. |
| `data` | `object` | No (default `{}`) | JSON-LD data payload to query against. |
| `ontology_url` | `string` | **Yes** | URL of the SOYA ontology endpoint (the service appends `/yaml` internally). |
| `sessionId` | `string` | No | UUID of an existing session. Omit or set to `null` to start a new session. |
| `epsilon` | `number` | No | Per-column ε value for this query. Must be a positive number. When omitted the server default (`EPSILON_BASE` env variable) is used. A higher value means more budget consumed but less noise; a lower value means less budget consumed but more noise. The global budget check still applies — the request is rejected with `BUDGET_EXHAUSTED` if the remaining budget is insufficient. |
| `adjusted_query` | `string` | No | An optional SPARQL query provided by the user. If provided, the service will bypass the LLM query generation step and execute this query directly, applying all subsequent validation and privacy steps. |

### Request Example

```json
{
  "question": "What is the average salary?",
  "data": {
    "@context": { "oyd": "https://soya.ownyourdata.eu/AnonymisationDemo2/" },
    "@graph": [
      {
        "@type": "oyd:Object1",
        "oyd:gehalt": 45000,
        "oyd:alter": 30,
        "oyd:geschlecht": "m"
      }
    ]
  },
  "ontology_url": "https://soya.ownyourdata.eu/AnonymisationDemo2",
  "sessionId": "c7f3b2a1-9d4e-4f8a-b6c1-2e5d7a3f9b0e",
  "epsilon": 0.05,
  "adjusted_query": "PREFIX oyd: <https://soya.ownyourdata.eu/AnonymisationDemo2/>\nSELECT (AVG(?gehalt) AS ?averageSalary) WHERE { ?s a oyd:Object1 ; oyd:gehalt ?gehalt . }"
}
```

### Success Response — `200 OK`

```json
{
  "response": "The average salary is approximately 45321.78.",
  "sessionId": "c7f3b2a1-9d4e-4f8a-b6c1-2e5d7a3f9b0e",
  "remainingPrivacyBudget": 0.9,
  "sessionEpsilonSpent": 0.1,
  "epsilonUsed": 0.05,
  "status": "success",
  "data": {
    "query_results": [
      { "averageSalary": 45321.78 }
    ],
    "sparql_query": "PREFIX oyd: <...> SELECT (AVG(?gehalt) AS ?averageSalary) WHERE { ... }"
  },
  "conversationHistory": [
    { "role": "user", "content": "What is the average salary?" },
    { "role": "assistant", "content": "The average salary is approximately 45321.78." }
  ]
}
```

### Success Response Schema

| Field | Type | Description |
|-------|------|-------------|
| `response` | `string` | Natural language answer generated by the LLM (numbers are validated against actual noisy values). |
| `sessionId` | `string` | UUID v4 identifying the session. Store this to continue multi-turn conversations. |
| `remainingPrivacyBudget` | `number` | Remaining global ε budget after this query. Reaches `0` when no more queries can be answered. |
| `sessionEpsilonSpent` | `number` | Cumulative ε consumed by this session so far. |
| `epsilonUsed` | `number` | The per-column ε value that was applied for this query. Equals the requested `epsilon` if one was provided, otherwise equals the server default (`EPSILON_BASE`). |
| `status` | `string` | `"success"` on a successful query. |
| `data` | `object` | Contains the raw query results and the generated SPARQL query. |
| `data.query_results` | `array<object>` | Noisy result rows. Each object maps column names to their noisy values (`number` or `string`). |
| `data.sparql_query` | `string` | The SPARQL query that was executed. |
| `conversationHistory` | `array<object>` | Full conversation history for the session. Each entry has `role` (`"user"` or `"assistant"`) and `content`. |

### Error Response — `200 OK` (pipeline error)

The service returns `200` with `"status": "error"` for **pipeline errors** (ontology fetch failed, query rejected, budget exhausted, etc.). This allows the frontend to always parse a consistent JSON shape.

```json
{
  "response": "Your question could not be answered due to privacy constraints.",
  "sessionId": "c7f3b2a1-9d4e-4f8a-b6c1-2e5d7a3f9b0e",
  "remainingPrivacyBudget": 0.9,
  "sessionEpsilonSpent": 0.0,
  "status": "error",
  "errorCode": "QUERY_REJECTED",
  "data": null,
  "conversationHistory": [
    { "role": "user", "content": "What is Anna's salary?" },
    { "role": "assistant", "content": "Your question could not be answered due to privacy constraints." }
  ]
}
```

### Error Response Schema (pipeline error)

| Field | Type | Description |
|-------|------|-------------|
| `response` | `string` | A user-safe error message suitable for display in the chat UI. |
| `sessionId` | `string` | Session UUID. |
| `remainingPrivacyBudget` | `number` | Remaining ε budget (unchanged if the query was rejected before budget deduction). |
| `sessionEpsilonSpent` | `number` | Cumulative ε spent by the session. |
| `status` | `string` | `"error"`. |
| `errorCode` | `string` | Machine-readable error code (see [Error Codes](#6-error-codes)). |
| `data` | `null` | Always `null` on error. |
| `conversationHistory` | `array<object>` | Conversation history including the error message as an assistant turn. |

### HTTP-Level Errors

| Status | Condition | Response Body |
|--------|-----------|---------------|
| `400 Bad Request` | Missing request body | `{ "error": "Request body is required", "status": "error" }` |
| `400 Bad Request` | Missing `question` field | `{ "error": "Question is required", "status": "error" }` |
| `400 Bad Request` | Missing `ontology_url` field | `{ "error": "Ontology URL is required", "status": "error" }` |
| `400 Bad Request` | Invalid `epsilon` (not a number or ≤ 0) | `{ "error": "epsilon must be a positive number", "status": "error" }` |
| `500 Internal Server Error` | Unhandled server exception | `{ "error": "An internal error occurred.", "status": "error" }` |

## 3. Get Privacy Budget — `GET /api/privacy-budget`

Returns the current global privacy budget status.

### Response — `200 OK`

```json
{
  "remaining_budget": 0.9,
  "epsilon_total": 1.0,
  "epsilon_base": 0.1
}
```

### Response Schema

| Field | Type | Description |
|-------|------|-------------|
| `remaining_budget` | `number` | How much ε budget is left. When this reaches `0`, no more queries can be served. |
| `epsilon_total` | `number` | The total ε budget (configured via `EPSILON_TOTAL` env variable). |
| `epsilon_base` | `number` | The ε cost per aggregate column per query (configured via `EPSILON_BASE` env variable). |

## 4. Reset Privacy Budget — `POST /api/privacy-budget/reset`

Resets the global privacy budget back to `epsilon_total`. **Intended for testing and demo purposes only.**

> [!CAUTION]
> Resetting the privacy budget in a production environment invalidates the differential privacy guarantee.

### Request

No request body required.

### Response — `200 OK`

```json
{
  "status": "success",
  "message": "Privacy budget reset.",
  "remaining_budget": 1.0
}
```

### Response Schema

| Field | Type | Description |
|-------|------|-------------|
| `status` | `string` | `"success"` |
| `message` | `string` | Human-readable confirmation. |
| `remaining_budget` | `number` | The budget after reset (equals `epsilon_total`). |

## 5. Common Response Types

### Conversation History Entry

```typescript
{
  role: "user" | "assistant";
  content: string;
}
```

### Query Result Row

Each row is a key-value object where keys are the SPARQL variable names and values can be `number` or `string`, depending on the query:

```typescript
{
  [columnName: string]: number | string;
}
```

Example rows:

```json
[
  { "averageSalary": 45321.78 },
  { "department": "Engineering", "headcount": 23, "avgSalary": 52000.50 }
]
```

## 6. Error Codes

These codes appear in the `errorCode` field of pipeline error responses from `/api/talk-to-data`:

| Error Code | User-Facing Message | Description |
|------------|---------------------|-------------|
| `ONTOLOGY_FETCH_FAILED` | "Unable to load the data schema. Please try again later." | The service could not download or parse the ontology from the provided URL. |
| `QUERY_GENERATION_FAILED` | "Unable to process your question. Please rephrase and try again." | The LLM failed to generate a SPARQL query from the question. |
| `QUERY_REJECTED` | "Your question could not be answered due to privacy constraints." | The generated SPARQL query violated one or more privacy rules (R1–R6). |
| `BUDGET_EXHAUSTED` | "The privacy budget for this session has been exhausted." | The remaining global ε budget is insufficient for this query. |
| `QUERY_EXECUTION_FAILED` | "An error occurred while running the query. Please try again." | SPARQL execution against the JSON-LD data failed. |

### Frontend Handling Recommendations

| Error Code | Suggested UI Behaviour |
|------------|------------------------|
| `ONTOLOGY_FETCH_FAILED` | Show a toast/alert; let user retry. May indicate a network issue or wrong URL. |
| `QUERY_GENERATION_FAILED` | Prompt the user to rephrase their question. |
| `QUERY_REJECTED` | Display the error message in the chat. The user's query touched sensitive data. |
| `BUDGET_EXHAUSTED` | Disable the input field and show a "budget exhausted" banner. Optionally offer a budget-reset button (demo mode only). |
| `QUERY_EXECUTION_FAILED` | Show a generic error; let user retry. |

## 7. Session Management

Sessions enable multi-turn conversations and per-session ε tracking.

### How Sessions Work

1. **First request**: Omit `sessionId` (or set to `null`). The service creates a new session and returns its UUID in `sessionId`.
2. **Subsequent requests**: Send the `sessionId` from the previous response to continue the same conversation.
3. **Invalid session ID**: If the provided `sessionId` is not found, a **new** session is created and a different `sessionId` is returned. The frontend should update its stored session ID accordingly.

### Session Data

Each session tracks:

| Property | Type | Description |
|----------|------|-------------|
| `session_id` | `string` | UUID v4 identifier. |
| `conversation_history` | `array<{ role, content }>` | All user and assistant messages in order. |
| `epsilon_spent` | `number` | Total ε consumed by queries in this session. |
| `created_at` | `string (ISO 8601)` | Timestamp of session creation. |

> [!IMPORTANT]
> Sessions are stored **in-memory** and are lost on server restart. The privacy budget is **global** — not scoped to individual sessions.

## 8. Frontend Integration Notes

### Recommended Request Flow

```
┌────────────────────────────────────────────────────────────────────┐
│  1. GET /api/ping          → verify server is up                  │
│  2. GET /api/privacy-budget → show initial budget in UI           │
│  3. POST /api/talk-to-data → send question, receive answer        │
│     └── repeat step 3 with the same sessionId for follow-ups      │
│  4. GET /api/privacy-budget → update budget display after query   │
└────────────────────────────────────────────────────────────────────┘
```

### Parsing Responses

```
if (response.status === "success") {
    // Display response.response in the chat bubble
    // Optionally show response.data.query_results in a table
    // Update budget indicator with response.remainingPrivacyBudget
} else if (response.status === "error") {
    // Display response.response as an error message in the chat
    // Use response.errorCode to decide on additional UI behaviour
}
```

### CORS

The Flask server does **not** currently include CORS headers. If the frontend is served from a different origin, a CORS middleware (e.g. `flask-cors`) will need to be added to the server.

### Key UI Elements to Implement

| Element | Data Source |
|---------|-------------|
| Chat message list | `conversationHistory` array |
| Budget indicator (progress bar) | `remainingPrivacyBudget` / `epsilon_total` from `/api/privacy-budget` |
| Per-session ε spent | `sessionEpsilonSpent` |
| Result data table | `data.query_results` (optional, for advanced view) |
| SPARQL query viewer | `data.sparql_query` (optional, for debug/advanced view) |
| Budget reset button | `POST /api/privacy-budget/reset` (demo mode only) |
| Input field disable | When `remainingPrivacyBudget` ≤ 0 or `errorCode === "BUDGET_EXHAUSTED"` |
